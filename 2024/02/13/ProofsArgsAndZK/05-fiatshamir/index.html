Chapter 5 Fiat-Shamir 转化

在交互式证明系统或论证系统中，V\mathcal{V}V 的掷币结果可向 P\mathcal{P}P 公开，这种协议称为公开掷币（public-coin）协议。不失一般性地，在这种协议中，可以约定 V\mathcal{V}V 向 P\mathcal{P}P 发送的所有信息均为掷币结果。这是因为 V\mathcal{V}V 的算法是确定性的，P\mathcal{P}P 可以自行计算 V\mathcal{V}V 原本的输出。

Fiat-Shamir 转化可将任何公开掷币协议转化为非交互式的、公开可验证（publicly verifiable）的协议 Q\mathcal{Q}Q。

随机谕示机模型

ROM（Random Oracle Model，随机谕示机模型）是一个理想化的模型，用以对目前发展的密码学杂凑函数进行建模，这类函数目前无法有效地与随机函数相区分。

随机函数 R:D→{0,1}κR:\mathcal{D}\to\{0, 1\}^\kappaR:D→{0,1}κ 的定义是，对于任意 x∈Dx\in \mathcal{D}x∈D，RRR 从 {0,1}κ\{0, 1\}^\kappa{0,1}κ 中均匀随机选取其对应的 R(x)R(x)R(x)。

在 ROM 模型中，V\mathcal{V}V 与 P\mathcal{P}P 均有随机谕示机的访问权限，每次向谕示机输入 xxx 时，谕示机在 {0,1}κ\{0, 1\}^\kappa{0,1}κ 中用一次独立的均匀随机确定 R(x)R(x)R(x) 并返回。后续询问同一个 xxx 时不进行随机，而是返回确定的 R(x)R(x)R(x)。这样的谕示机可以视为对随机函数 RRR 的谕示机。

对于一个随机函数 RRR，指定 RRR 需要 ∣D∣⋅κ|\mathcal{D}|\cdot \kappa∣D∣⋅κ 位，而 D\mathcal{D}D 在实际设置中为保证安全性一般设置为较大的集合，因此直接指定 RRR 并不实际。往往将随机谕示机替换为某一个确定的杂凑函数，如 SHA-3，其存在简短描述（布尔电路、程序）。即便一个协议在 ROM 模型下是安全的，攻击者仍可通过攻击替换谕示机的杂凑函数破坏协议的安全性，但在使用足够安全的杂凑函数时可以认为协议仍然是安全的。

Fiat-Shamir 转化

直接使用随机谕示机 RRR 产生 V\mathcal{V}V 的所有消息并不能保证 soundness，因为 P\mathcal{P}P 可以在未选定输出的情况下预测 V\mathcal{V}V 的消息。因此 V\mathcal{V}V 的消息必须在 P\mathcal{P}P 确定其本轮的输出后才能交由 RRR 生成，因此对于 RRR 的查询输入需要包含 P\mathcal{P}P 的输出。这自然引出如下设计：

  • P\mathcal{P}P 发送消息 g1g_1g1​
  • 生成 V\mathcal{V}V 的消息 r1=R(x,g1)r_1=R(x, g_1)r1​=R(x,g1​)
  • P\mathcal{P}P 发送消息 g2g_2g2​
  • 生成 V\mathcal{V}V 的消息 r2=R(x,g1,g2)r_2=R(x, g_1, g_2)r2​=R(x,g1​,g2​)
  • …\dots…
  • 生成 V\mathcal{V}V 的消息 ri=R(x,g1,…,gi)r_i=R(x, g_1, \dots, g_i)ri​=R(x,g1​,…,gi​)

如此一来，P\mathcal{P}P 只需要发送完整的 transcript（交互记录）即可。

一般可以使用 hash-chaining 的方法，即每轮的消息 rir_iri​ 仅由

ri=R(x,i,ri−1,gi)r_i=R(x, i, r_{i-1}, g_i) ri​=R(x,i,ri−1​,gi​)

生成，这保证了 RRR 的输入的简短性，并且在 ROM 模型下仍然可以被证明是安全的。

当攻击方可以选择公共输入 xxx 时，Fiat-Shamir 转化存在一个常见漏洞，即在 rir_iri​ 的生成中不包含 xxx。包含 xxx 的版本称为强 Fiat-Shamir 转化，不包含 xxx 的版本称为弱 Fiat-Shamir 转化。

例如，假设在 GKR 协议中，希望证明 C(x)=y\mathcal{C}(x)=yC(x)=y，由于在最后 V\mathcal{V}V 才需要验证 x~\tilde xx~ 的一个点值，如果 xxx 不被包含在消息 rir_iri​ 的生成中，攻击方可以选择任意一个使得 x~(r)\tilde x(r)x~(r) 与上一轮的结果吻合的 xxx，这样的 xxx 可以在线性时间内得到，而 V\mathcal{V}V 无法检验出这种不一致性，此时协议并不可靠。

简言之，为避免公共输入被替换，需要将例如 C,x,y\mathcal{C}, x, yC,x,y 这样的信息包含在随机函数的查询输入中。

安全性

设 I\mathcal{I}I 的通信轮次为 ttt，应用 Fiat-Shamir 转化后得到协议 Q\mathcal{Q}Q。若 Q\mathcal{Q}Q 下证明方 P\mathcal{P}P 可在 TTT 时间内产生 xxx 的论证，使得 V\mathcal{V}V 以概率 ε\varepsilonε 接受该论证，则 P\mathcal{P}P 可转化为协议 I\mathcal{I}I 下的 P′\mathcal{P}'P′，其以至少 (ε/T)O(t)(\varepsilon/T)^{\mathcal{O(t)}}(ε/T)O(t) 的概率使得 V\mathcal{V}V 接受 xxx。

假设原本的 soundness error 不超过 ε\varepsilonε，上述事实表明对于 Q\mathcal{Q}Q 而言，其 soundness error 保证仅为 TεO(1/t)T\varepsilon^{\mathcal{O}(1/t)}TεO(1/t)，而这是不可忽略的。

如果语言 L\mathcal{L}L 的 IP I\mathcal{I}I 具备轮可靠性（round-by-round soundness），那么其对应的 Q\mathcal{Q}Q 在 ROM 模型下具备可靠性。轮可靠性的定义：

  • 对于 transcript 的集合，存在正常 transcript 与损坏 transcript 的划分
  • 若 I\mathcal{I}I 的某一轮中 transcript 已处于损坏状态，则后续轮中 transcript 必处于损坏状态
  • 如果 x∉Lx\not\in \mathcal{L}x∈L，则 transcript 一开始便处于损坏状态
  • 对于损坏的 transcript，V\mathcal{V}V 最终可以检验出错误并拒绝 xxx

简而言之，轮可靠性使得 P\mathcal{P}P 无法通过回滚状态，尝试不同输出来使得 RRR 恰好给出令 V\mathcal{V}V 接受的随机值。

安全等级

IP（交互式证明系统）的安全等级由其 soundness error（可靠性误差）的负对数定义。例如一个 δs≤2−128\delta_s\le 2^{-128}δs​≤2−128 的 IP 被称为具有 128bit 信息论安全等级（statistical security）。

类似地，对于非交互式协议 Q\mathcal{Q}Q，可定义计算安全性。若 P\mathcal{P}P 为假命题生成合法证明所需的时间开销为 TTT，则称 Q\mathcal{Q}Q 具有 (log⁡T)(\log T)(logT)-bit 计算安全等级（computational security）。

对于交互式论证系统而言，一般由其 soundness error 定义其安全等级，称为交互安全等级（interactive security）。

对于交互式系统，部署时所需的安全等级一般比较低。即便安全等级只有 60bit，经过 2302^{30}230 交互，P\mathcal{P}P 也只有大约 2−302^{-30}2−30 的概率向 V\mathcal{V}V 证明一个假命题。

而对于非交互式系统，一般需要较高的安全等级，这是因为其安全性本质上依赖于杂凑函数的安全性。截至本笔记的更新时间，bitcoin 的 hashrate 峰值大约是 700 EH/s，即每秒大约计算 7×10207\times 10^{20}7×1020 次 SHA256。以这个速度，计算 2802^{80}280 次 SHA256 仅需不到半个小时。

常数轮次协议在 ROM 下的安全性

若 I\mathcal{I}I 是常数轮次、公开掷币的交互式证明系统或交互式论证系统，且 I\mathcal{I}I 的可靠性误差可忽略，则应用 Fiat-Shamir 转化得到的 Q\mathcal{Q}Q 的计算可靠性误差亦可忽略。亦即 P\mathcal{P}P 无法在多项式时间内在 Q\mathcal{Q}Q 中生成一个假命题的证明。

证明思路主要使用了反证法。

设 PFS\mathcal{P_{FS}}PFS​ 是 Q\mathcal{Q}Q 中的证明策略，其运行时间为 TTT，为 x∉Lx\not\in\mathcal{L}x∈L 生成合法证明的概率为 ε\varepsilonε，则存在一个协议 I\mathcal{I}I 中的策略 P∗\mathcal{P}^\astP∗，其可在 I\mathcal{I}I 中以至少 ε∗=Ω(ε/T)\varepsilon^\ast=\Omega(\varepsilon/T)ε∗=Ω(ε/T) 概率证明 xxx。

这个事实的证明比较容易，P∗\mathcal{P}^\astP∗ 在与 V\mathcal{V}V 交互时，只需要同时充当 Q\mathcal{Q}Q 中应答 PFS\mathcal{P_{FS}}PFS​ 的查询的谕示机即可。由于 V\mathcal{V}V 的输出来源于掷币，其完全满足随机谕示机的性质，P∗\mathcal{P}^\astP∗ 可直接用 V\mathcal{V}V 的输出替代 Q\mathcal{Q}Q 中随机谕示机的输出，不破坏 PFS\mathcal{P_{FS}}PFS​ 为 xxx 生成证明的概率。本质上 P∗\mathcal{P}^\astP∗ 只需要做 PFS\mathcal{P_{FS}}PFS​ 与 V\mathcal{V}V 的中间人即可。

画蛇添足一下，这里 −log⁡ε-\log \varepsilon−logε 实际上就是 Q\mathcal{Q}Q 的计算安全性等级（是否含 log⁡T\log TlogT 项取决于定义），因为如果单次的概率是 ε\varepsilonε，明显 PFS\mathcal{P_{FS}}PFS​ 生成合法证明的期望时间是 T/εT/\varepsilonT/ε，log⁡(T/ε)=−log⁡ε+log⁡T\log (T/\varepsilon)=-\log \varepsilon+\log Tlog(T/ε)=−logε+logT。

记 PFS\mathcal{P_{FS}}PFS​ 生成的第一个消息为 g1g_1g1​，PFS\mathcal{P_{FS}}PFS​ 在确定 g1g_1g1​ 前可能尝试调用 O(T)\mathcal{O}(T)O(T) 次谕示机求 r1=R(x,g1)r_1=R(x, g_1)r1​=R(x,g1​)，P∗\mathcal{P}^\astP∗ 只需要随机选择其中一个 g1g_1g1​ 发送给 V\mathcal{V}V 即可。如果 PFS\mathcal{P_{FS}}PFS​ 在 O(T)\mathcal{O}(T)O(T) 之内可以以 ε\varepsilonε 概率试出其所需的 r1r_1r1​，那么 P∗\mathcal{P}^\astP∗ 随机选择一个 g1g_1g1​，正确的概率明显至少 Ω(ε/T)\Omega(\varepsilon/T)Ω(ε/T)。具体细节如下：

  • 注意 P∗\mathcal{P}^\astP∗ 开始运行需要 PFS\mathcal{P_{FS}}PFS​ 给出 g1g_1g1​，而最终的 transcript 中的 r1r_1r1​ 必然由谕示机根据最终的 g1g_1g1​ 确定，因此可设 PFS\mathcal{P_{FS}}PFS​ 必定对其最终选择的 g1g_1g1​ 查询 r1=R(x,g1)r_1=R(x, g_1)r1​=R(x,g1​)。不失一般性，可假设 PFS\mathcal{P_{FS}}PFS​ 使用的查询次数恰好为 TTT，且不做重复查询。

  • 因此 P∗\mathcal{P}^\astP∗ 可随机选择 TTT 以内的整数 iii，运行 PFS\mathcal{P_{FS}}PFS​ 至其第 iii 次查询 RRR，对于前面的查询 P∗\mathcal{P}^\astP∗ 回应以自身生成的随机值，若查询形如 R(x,g1)R(x, g_1)R(x,g1​)，则将 g1g_1g1​ 作为发送给 V\mathcal{V}V 的消息，并将 V\mathcal{V}V 的回复作为谕示机的返回值。不难发现这不影响 PFS\mathcal{P_{FS}}PFS​ 对于后续量的生成与查询。
